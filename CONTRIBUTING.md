# Contributing and Feedback

## Current project status

ProChat Memory for QA is in a private beta and public evaluation phase. The public source snapshot is source-available rather than open source.

Approved beta testers may evaluate the designated beta version only under the applicable provisional evaluation terms. Those rights do not permit unrestricted production use, resale, redistribution, sublicensing, public SaaS operation, embedding in another commercial product, or competitive commercial use.

Production and commercial use require a separate written agreement.

MCP is not included in the public source snapshot.

## Feedback channels

GitHub Issues and Discussions in the public repository are the initial public feedback channels.

Useful feedback includes:

- installation problems;
- command behavior;
- documentation clarity;
- privacy or safety concerns;
- compatibility observations;
- workflow friction;
- sanitized defect reports;
- suggestions for clearer examples or terminology.

## Code contributions

Code contributions are not accepted during this beta phase.

The private canonical repository remains the source of truth. Product fixes and implementation changes are made there, reviewed, validated, and included in a later generated public snapshot.

Do not open source-code pull requests during the beta unless the maintainers explicitly change this policy.

## Documentation suggestions

Sanitized documentation suggestions are welcome through GitHub Issues and Discussions in the public repository.

A useful documentation report should identify:

- the document and section;
- what was unclear or inaccurate;
- the expected wording or behavior;
- a sanitized example where helpful.

Minor documentation pull requests may be accepted later under a separately announced contribution policy.

## Sensitive information

Never submit any of the following through Issues, Discussions, pull requests, comments, screenshots, or attachments:

- passwords, tokens, credentials, keys, or certificates;
- client or customer data;
- private QA memory;
- confidential logs or traces;
- private screenshots;
- internal URLs, hostnames, or repository links;
- proprietary project names;
- personal information;
- unredacted workspace files;
- sensitive database contents;
- confidential business or security information.

Use synthetic or thoroughly sanitized examples.

## Feedback handling

Public feedback is reviewed and converted into a sanitized internal task when action is required.

The fix or documentation change is implemented privately, validated through the public-export process, and included in a later generated public snapshot. The public report may then be updated with the version containing the change.

## Licensing and contributor rights

Source visibility does not grant permission to use, modify, redistribute, host, embed, or commercialize the software beyond the rights provided by the applicable beta agreement, evaluation terms, source-available license, or commercial agreement.

Substantive external code contributions remain disabled until contributor licensing, copyright, patent, and relicensing terms are formally established.
